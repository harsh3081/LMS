# ADR-003: Single Database, Shared Schema with Row-Level Location Scoping for Multi-Outlet Tenancy

## Status

Accepted — 2026-07-15

## Context

The BRD requires that LMS be able to scale from a single dealership to a multi-outlet, multi-location dealer group "without redesign," and that SM/GM roles be able to obtain centralized reporting across outlets/locations within a dealer group. Two alternative tenancy models were evaluated: schema-per-outlet (a separate PostgreSQL schema per dealership location) and database-per-tenant (a fully isolated database per dealer group or outlet). Schema-per-outlet was rejected because it breaks the centralized cross-outlet reporting that SM/GM explicitly needs, forcing cross-schema query fan-out or a separate reporting-aggregation layer. Database-per-tenant was rejected as over-engineered for dealership scale — it multiplies operational overhead (migrations, backups, connection pooling) for a data volume that does not warrant full isolation.

## Decision

We will use a single database with a shared schema across all tenants. Every core entity (leads, enquiries, bookings, users, audit_log, etc.) carries a mandatory `location_id` column, and a mandatory `dealer_group_id` column where cross-location grouping is meaningful. Row-level scoping — restricting every query to the caller's authorized `location_id`/`dealer_group_id` — is enforced centrally in the data-access layer (e.g. a shared repository base class or ORM-level query interceptor), not left to be repeated ad hoc in each endpoint. A single standalone dealership is represented simply as one `location_id` row under one `dealer_group_id`; centralized group reporting for SM/GM becomes a straightforward cross-location SQL query rather than a cross-database or cross-schema federation problem.

## Consequences

**Positive**: Scaling from one dealership to a multi-outlet dealer group requires no schema redesign or data migration between models — it is purely a matter of adding more `location_id` rows; SM/GM centralized cross-outlet reporting is a simple, performant same-database query; operational overhead (one set of migrations, one backup/restore procedure, one connection pool) stays low and matches dealership-scale needs.

**Negative**: Row-level scoping enforcement in the data-access layer becomes a single point of failure for tenant isolation — a bug or a bypassed code path that omits the `location_id` filter risks cross-location data leakage, so this enforcement point requires dedicated test coverage and code review scrutiny; a single noisy or misbehaving tenant/location can in principle affect performance for all other tenants sharing the same database instance.

**Neutral**: All new tables added in future iterations must include `location_id` (and `dealer_group_id` where applicable) as a first-class column from the outset, and all new repository/query code must route through the shared scoping mechanism rather than querying tables directly.
