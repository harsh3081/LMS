# ADR-007: Provider-Agnostic Notification Abstraction with Email, SMS, and WhatsApp Adapters

## Status

Accepted — 2026-07-15

## Context

Unlike the Booking/Invoicing/Payments integrations, which are explicitly deferred, notifications are a live Phase-1 requirement: DSEs must be able to share brochures and quotations, and the system must send due-follow-up, test-drive, and reassignment notifications through multiple channels. The target market is primarily India-based dealership operations, where WhatsApp is a commonly used customer-communication channel alongside email and SMS. Committing directly to a single vendor's SDK throughout the codebase would make future provider changes (due to cost, reliability, or regional coverage) expensive, and WhatsApp Business API usage carries a known onboarding constraint: message templates must be pre-approved by the platform/provider before they can be sent, which is a schedule risk if not planned for early.

## Decision

We will implement a provider-agnostic notification abstraction inside the LMS backend — a single internal interface (e.g. `NotificationService.send(channel, template, recipient, payload)`) — with concrete adapters plugged in behind it: an email adapter using SMTP or a transactional email provider (e.g. Amazon SES or equivalent), and a combined SMS + WhatsApp adapter using a unified messaging gateway (e.g. Twilio, or a regional provider such as Gupshup or MSG91 given the India dealership market). Application code (follow-up reminders, test-drive notifications, reassignment alerts, brochure/quotation sharing) will call the abstraction, not the underlying provider SDKs directly.

## Consequences

**Positive**: The abstraction isolates the rest of the codebase from provider-specific SDKs and payload formats, so switching or adding a provider (e.g. moving from one WhatsApp gateway to another, or adding a regional SMS provider) is a contained adapter change rather than a codebase-wide rewrite; having distinct channel adapters lets email, SMS, and WhatsApp evolve independently (e.g. different retry policies, different rate limits) behind a common interface.

**Negative**: WhatsApp Business API template pre-approval is an external dependency with its own lead time and is a known project risk — if templates are not submitted and approved early, WhatsApp-channel notifications will not be available at launch even though the code is ready; maintaining an abstraction layer over three channels adds an initial implementation and testing cost compared to calling a single provider's SDK directly.

**Neutral**: Notification delivery status (sent/delivered/failed) and retry behavior must be defined per channel as part of the adapter contract, since email, SMS, and WhatsApp gateways report delivery status differently; the specific vendor selected for SMS/WhatsApp (Twilio vs. Gupshup vs. MSG91) is a configuration/procurement decision to be finalized during implementation and does not change this ADR's architectural commitment to the abstraction.
