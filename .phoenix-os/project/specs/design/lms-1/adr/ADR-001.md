# ADR-001: Use a React (Vite) SPA Frontend with a NestJS/TypeScript Backend

## Status

Accepted — 2026-07-15

## Context

The Lead & Enquiry Management Web Application (LMS) is a standalone, API-first web app for automotive dealerships serving a 3-tier role hierarchy (DSE → TL → SM/GM). The BRD requires that DSEs be able to open and act on lead/enquiry task detail without a full page reload, which calls for a rich client-side interaction model rather than server-rendered page navigation. Separately, the BRD mandates that all core functions be exposed via well-documented REST APIs, and that the web application itself consume those same APIs rather than have a privileged internal path — an API-first constraint that shapes both the backend framework choice and the contract-generation tooling around it. The dealership's IT organization is small, so the technology choice must minimize the number of languages and frameworks a limited team has to maintain.

## Decision

We will build the LMS frontend as a React single-page application scaffolded and bundled with Vite, and the backend as a Node.js service written in TypeScript using the NestJS framework. NestJS was chosen over lighter alternatives (e.g. Express, Fastify with hand-rolled structure) because of its first-class OpenAPI/Swagger generation, its opinionated modular architecture (controllers/services/modules), and its built-in support for guards, interceptors, and DTO validation — all of which reduce boilerplate for the REST-API-first mandate. Using TypeScript on both frontend and backend gives the team a single language across the stack.

## Consequences

**Positive**: A single language (TypeScript) across frontend and backend lowers onboarding and context-switching cost for a small dealership IT team; React's component model and client-side routing directly satisfy the no-page-reload task-detail requirement; NestJS's native OpenAPI decorators keep the REST contract (see ADR-006) in sync with the implementation with minimal extra authoring effort.

**Negative**: NestJS's dependency-injection and decorator-heavy style has a steeper initial learning curve than a minimal Express app, which may slow the first few sprints for developers unfamiliar with the framework; a SPA introduces client-side build tooling (Vite), bundle-size management, and client-side routing/state concerns that a traditional server-rendered app would avoid.

**Neutral**: All future modules (Booking, Invoicing, Payments) that integrate with LMS will need to either adopt the same REST/OpenAPI contract style or go through an adapter, since the web client has no privileged access path into the backend beyond the published API.
