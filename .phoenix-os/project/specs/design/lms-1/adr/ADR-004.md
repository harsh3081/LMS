# ADR-004: Use Stateful Server-Side Sessions for Phase-1 Authentication

## Status

Accepted — 2026-07-15

## Context

The BRD commits LMS to standalone login in Phase 1, with centralized SSO/OAuth2 explicitly deferred to a future phase (that phase deferral is a given BRD constraint and is not itself the subject of this decision). This ADR addresses the specific Phase-1 implementation mechanism for authenticating DSE/TL/SM/GM users against the standalone LMS service. The mechanism must support straightforward, immediate session revocation (e.g. on password change, account deactivation, or role change by an admin) and must not introduce dependencies on infrastructure the standalone deployment does not otherwise require (ADR-005). It should also avoid locking the system into an approach that makes the future SSO/OAuth2 migration disruptive to business logic.

## Decision

We will implement Phase-1 authentication using stateful server-side sessions, issued via HTTP-only, Secure, SameSite cookies, backed by password hashing with bcrypt or Argon2, and an idle-timeout auto-logout policy. Session state will be held behind a thin authentication/authorization abstraction (an internal auth module with a stable interface for "who is the current user and what can they do") so that the concrete session mechanism is swappable. Future OAuth2/OIDC-based SSO integration will be introduced by implementing a new strategy behind this same abstraction, without requiring changes to business logic that depends on the current-user context.

## Consequences

**Positive**: Server-side sessions give trivial, immediate revocation (deleting the session record instantly invalidates the credential), which is important for admin-initiated deactivation and role changes; this is the simplest correct mechanism for a single standalone service with no cross-service identity federation requirement in Phase 1; the thin auth abstraction isolates the rest of the codebase from the specific mechanism, keeping the future SSO/OAuth2 migration a contained, additive change.

**Negative**: Server-side session storage introduces server-side state (a session store) that must be sized, backed up, and — if the service is ever horizontally scaled — shared across instances (e.g. via the database or a shared cache), which is a deployment consideration not present with a fully stateless token approach; session-cookie-based auth requires care around CSRF protections for state-changing requests, in addition to the XSS mitigations already needed for cookie-based credentials.

**Neutral**: Password hashing (bcrypt/Argon2) and idle-timeout policy parameters (timeout duration, sliding vs. absolute expiry) must be defined and documented as part of the security baseline; when SSO/OAuth2 is introduced in a future phase, session-based auth may be retained as a fallback or replaced entirely — that decision is out of scope for this ADR and will be recorded separately when the SSO phase is scoped.
