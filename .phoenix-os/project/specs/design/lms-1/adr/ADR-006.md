# ADR-006: OpenAPI 3.1 as the REST Contract of Record, with Transactional-Outbox Webhooks for Event Delivery

## Status

Accepted — 2026-07-15

## Context

The BRD mandates that all core LMS functions be exposed via well-documented REST APIs, and separately requires that key business events (e.g. "EnquiryConverted") be published starting in Phase 1, even though no live subscriber exists yet — this is explicitly to support future integration with the Booking, Invoicing, and Payments modules without those modules requiring direct database access into LMS. Two concerns therefore need a decision: what format the REST contract itself is documented and published in, and what mechanism guarantees that business events are reliably emitted without losing them, without introducing infrastructure (such as a full message broker) that Phase-1 scale does not justify.

## Decision

We will use OpenAPI 3.1 as the source-of-truth contract format for all LMS REST APIs, generated from NestJS's native OpenAPI decorators (ADR-001) and published as the artifact future module owners (Booking, Invoicing, Payments) will consume to integrate. For business events, we will use outbound webhooks delivered via a transactional-outbox pattern: each business event (e.g. `EnquiryConverted`, `LeadReassigned`) is persisted as a row in an outbox table within the same database transaction as the business state change, and a background worker polls the outbox table and dispatches the corresponding webhook calls, marking entries delivered or retrying on failure.

## Consequences

**Positive**: OpenAPI 3.1 is a widely-adopted, tool-supported standard, making it straightforward for future module owners to generate client SDKs or contract tests against LMS without bespoke documentation; the transactional-outbox pattern guarantees at-least-once event delivery with no risk of an event being silently dropped because the business transaction and the event-persistence step are atomic; this achieves reliable event publishing without standing up a message broker (e.g. Kafka/RabbitMQ) that Phase-1 scale and subscriber count do not justify.

**Negative**: The outbox pattern requires a background dispatch worker, retry/backoff logic, and idempotency handling on the (currently nonexistent) subscriber side once one is introduced, which is additional implementation surface for a feature with no live consumer yet; at-least-once delivery means downstream consumers must be designed to handle duplicate event deliveries.

**Neutral**: If event volume or subscriber count grows significantly in a later phase, the outbox-to-webhook dispatch mechanism may need to be replaced by a message broker — that would be a superseding decision addressed in a future ADR rather than a change to this one; the OpenAPI spec becomes a build artifact that must be kept in CI as a published, versioned document for consumers to reference.
