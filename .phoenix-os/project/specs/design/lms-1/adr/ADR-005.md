# ADR-005: Containerized Deployment to a Managed Container Service with Managed PostgreSQL

## Status

Accepted — 2026-07-15

## Context

The BRD requires LMS to be independently and standalone deployable — it must not require the future Booking, Invoicing, or Payments modules to exist or be reachable — and it commits to a 99.5% uptime SLA during business hours. The system must also be able to scale from a single dealership to a multi-outlet dealer group (ADR-003) without a redesign of how it is deployed. The dealership's IT organization is small and does not have dedicated database administration capacity, which weighs against self-managed infrastructure for the stateful components.

## Decision

We will package the LMS application (frontend build artifacts served alongside, or independently of, the NestJS backend — see ADR-001) as Docker containers and deploy them to a managed container service (e.g. a managed Kubernetes offering, ECS/Fargate, or an equivalent managed-container PaaS) within a single cloud region. The primary data store (ADR-002) will run as a managed PostgreSQL instance (e.g. a cloud provider's managed PostgreSQL service) rather than a self-hosted database server.

## Consequences

**Positive**: Containerization guarantees a reproducible, standalone deployment that does not require any other module (Booking/Invoicing/Payments) to be present, directly satisfying the independent-deployability requirement; a managed container service and managed PostgreSQL instance carry the 99.5% uptime target largely through the provider's operational tooling (health checks, automated failover, automated backups) rather than requiring the dealership to build or staff that capability; scaling to a multi-outlet dealer group is a matter of resizing container replica counts and database instance tier, not redesigning the deployment topology.

**Negative**: A single-region deployment means the SLA is bounded by that region's availability characteristics — a regional outage affects the entire application, and cross-region disaster recovery is not covered by this decision and would need a separate ADR if required in the future; reliance on managed services introduces a degree of cloud-provider lock-in and recurring managed-service cost compared to self-hosting.

**Neutral**: The team must define container build, image registry, and deployment pipeline conventions (covered under devops tooling) as part of implementation, and must document the chosen region and its published SLA against the 99.5% business-hours uptime commitment.
