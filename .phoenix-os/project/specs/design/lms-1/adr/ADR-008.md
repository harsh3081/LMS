# ADR-008: Object Storage for Files with Server-Side HTML-to-PDF Rendering and a Spreadsheet Library for Excel Export

## Status

Accepted — 2026-07-15

## Context

LMS must support sharing brochures with prospects, generating and sharing quotations, and exporting reporting data (leads, enquiries, funnel/leaderboard data) to Excel. Brochures are pre-existing binary assets; quotations are dynamically generated documents whose layout is likely to change over time as business requirements evolve; and Excel exports are generated on demand from relational data (ADR-002). Storing large binary assets directly in the primary PostgreSQL database would bloat the transactional database, complicate backups, and work against the JSONB/relational design established in ADR-002. Quotation layout changes should not require a backend code deployment every time marketing or sales wants to adjust formatting.

## Decision

We will store brochures and generated artifacts (rendered quotation PDFs, exports) in an S3-compatible object storage bucket, accessed via signed, time-limited URLs for sharing rather than serving files through the application server directly. Quotations will be generated using server-side HTML-to-PDF rendering — an HTML template (populated with quotation data) rendered to PDF by a server-side rendering engine — rather than a code-driven PDF-drawing library, so that layout changes are template edits rather than code changes. Excel exports will be produced using a standard spreadsheet-generation library operating on data queried from PostgreSQL.

## Consequences

**Positive**: Keeping large binaries out of the primary database keeps PostgreSQL backups and replication focused on transactional data, improving both performance and operational simplicity; signed links from object storage give a natural, secure mechanism for brochure/quotation sharing (e.g. via the notification channels in ADR-007) without proxying file bytes through the application server; HTML-templated PDF generation means quotation layout and branding changes can be made by editing a template without a backend redeploy.

**Negative**: Server-side HTML-to-PDF rendering (e.g. via a headless browser or rendering library) has a nontrivial memory/CPU footprint per render and needs to be sized and rate-limited so that a burst of quotation requests does not degrade the application server; signed URLs need an expiry and access-control policy design so that shared links do not remain valid indefinitely or leak access to unauthorized parties.

**Neutral**: Object storage lifecycle policies (retention, cleanup of stale generated artifacts) must be defined as part of implementation; the specific spreadsheet library and PDF-rendering engine are implementation-level choices to be finalized during technical design and do not change this ADR's architectural commitment to object storage plus server-side templated rendering.
