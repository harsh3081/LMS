# ADR-002: Use a Single Managed PostgreSQL Instance as the Primary Data Store

## Status

Accepted — 2026-07-15

## Context

LMS must guarantee relational integrity and transactional consistency in two critical areas: test-drive slot booking, where a race between two DSEs must never result in a double-booking of the same slot, and the audit trail of lead/enquiry reassignments, which must remain consistent even under concurrent updates. The dashboard, funnel, and leaderboard reporting requirements for TL/SM/GM roles depend heavily on SQL-style aggregation (counts, conversion rates, rankings) across leads, enquiries, and activities. At the same time, the BRD calls for configurable mandatory fields on leads and enquiries per dealership, which implies some schema flexibility without wanting to stand up a second, separate datastore purely for that purpose.

## Decision

We will use PostgreSQL 16 (or later) as the single primary relational data store for LMS, deployed as one managed instance per environment. Transactional/relational tables will hold core entities (leads, enquiries, users, bookings, audit_log), with row-level locking or `SELECT ... FOR UPDATE`/unique-constraint patterns used to prevent double-booking of test-drive slots. Configurable mandatory fields will be modeled using PostgreSQL's native JSONB columns rather than a second document store, keeping the entire domain in one engine.

## Consequences

**Positive**: Full ACID compliance directly satisfies the no-double-booking and audit-consistency requirements with no additional compensating controls; native SQL aggregation serves dashboard, funnel, and leaderboard reporting without an auxiliary analytics engine; JSONB support absorbs configurable-field variability without a second datastore or its associated operational overhead.

**Negative**: All configurable-field queries and validations that need to filter or index on JSONB contents require deliberate GIN indexing and query design, or they will degrade in performance as data volume grows; vertical scaling of a single PostgreSQL instance has a practical ceiling and will need to be revisited (e.g. read replicas, partitioning) if data volume or write concurrency significantly exceeds dealership-scale projections.

**Neutral**: Database migrations must be authored and versioned through a formal migration tool (e.g. TypeORM/Prisma migrations or Flyway) rather than ad-hoc SQL scripts, since a single shared schema across all tenants (see ADR-003) makes migration discipline mandatory rather than optional.
