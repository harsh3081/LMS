# Issue #36 — Prevent Double-Booking of Demo Vehicles

Fast-track implementation (no separate spec.md/tech-design.md/todo.md — direct TDD implementation against the existing #34/#35 foundation and `.phoenix-os/project/specs/design/lms-1/adr/ADR-002.md`, per orchestrator instruction). This document records the design decisions, the pg-mem investigation that shaped them, test results, and known gaps.

Builds directly on `.phoenix-os/project/specs/34/NOTES.md` (`demo_vehicles`/`test_drives` schema, `TestDrivesRepository`/`TestDrivesService`/`TestDrivesController`) and `.phoenix-os/project/specs/35/NOTES.md` (`findBookedInRange`, the scheduler grid, `schedulerGrid.ts`). Those Stories deliberately left `TestDrivesService.book()` accepting bookings naively — this Story owns closing that gap entirely.

## Design decisions

### Concurrency mechanism — app-level pre-check + DB-level partial UNIQUE index, NOT `SELECT ... FOR UPDATE`

ADR-002 mentions both "row-level locking" and "unique-constraint patterns" as acceptable. Before committing to either, a throwaway sandbox investigation (three disposable Jest specs, deleted after use) established the following, run against this backend's actual pg-mem test harness (`test/support/test-data-source.ts`):

1. **`SELECT ... FOR UPDATE` is accepted syntactically by pg-mem but provides no real locking.** Two "concurrent" `dataSource.transaction()` calls, each doing `SELECT ... FOR UPDATE` → read → `await` a delay → conditional `INSERT`, were fired via `Promise.allSettled`. The interleaving log showed **both** transactions completed their `SELECT` (reading 0 conflicting rows) **before either committed**, and **both** inserts succeeded — i.e. two logically-racing "check-then-insert" transactions produced a duplicate row despite the `FOR UPDATE` clause. This is a real limitation of pg-mem (a synchronous, single-threaded, in-memory SQL engine substituting for real Postgres in this sandbox — there is no Docker/Postgres install available here, see `test/support/test-data-source.ts`'s own comment), not a flaw in the row-locking *concept* — real Postgres would serialize these correctly. But it means `FOR UPDATE` **cannot be relied on as the primary mechanism in THIS test harness**, and this Story's own test suite runs exclusively against that harness.
2. **A plain `UNIQUE` constraint, by contrast, correctly rejected the second of two identically-raced inserts** in the exact same throwaway harness — `INSERT` #2 failed with a genuine Postgres-shaped `QueryFailedError` (`duplicate key value violates unique constraint`, SQLSTATE `23505`), even though both transactions had already both "passed" their own read step before either committed. This is because pg-mem *does* correctly enforce UNIQUE/CHECK constraints at write time (verified separately — see below) — it just doesn't implement real transactional row-level locking/serialization for `SELECT ... FOR UPDATE`.
3. A **true Postgres range-overlap `EXCLUDE USING gist`** constraint (the textbook-correct DB-level guarantee for AC4's "overlapping, not just exact matches") was attempted and confirmed **not usable in this sandbox**: `CREATE EXTENSION IF NOT EXISTS btree_gist` fails with `Extension does not exist` — pg-mem does not implement the `btree_gist` extension GiST exclusion constraints depend on.

**Conclusion / what was actually built**, mirroring this codebase's own existing precedent for a structurally identical problem (`EnquiriesService.convert`'s Lead→Enquiry conversion: an app-level pre-check inside a transaction, backed by `enquiries`' `UNIQUE(lead_id)` DB constraint, migration `1700000000003-CreateEnquiries.ts`):

- **PRIMARY mechanism (AC1/AC4, general range-overlap correctness):** `TestDrivesService.assertNoConflict` runs an app-level query — **reusing `TestDrivesRepository.findBookedInRange` verbatim**, passed the *same* transactional `EntityManager` the subsequent insert uses — before the insert, inside `dataSource.transaction(...)`. This correctly rejects exact duplicates, partial overlaps in either direction, and full-containment overlaps (proven by the conflict-matrix tests below), and correctly does NOT reject adjacent/non-overlapping slots, different vehicles, or different tenants.
- **BACKSTOP mechanism (AC3, the raced-request case the app-level check alone cannot always catch):** migration `1700000000015` adds a **partial** UNIQUE index — `UNIQUE (vehicle_id, slot_start, slot_end) WHERE status = 'Booked'` — and `book()` catches a `QueryFailedError` with `code === '23505'` (the actual Postgres SQLSTATE for `unique_violation`, confirmed identical whether raised by pg-mem or real Postgres — TypeORM surfaces the driver's real code, not a synthetic one) around the insert, translating it into the same `TestDriveSlotConflictError` the pre-check throws, computing suggestions freshly rather than leaking a raw 500.
- **Why PARTIAL (not table-wide) UNIQUE**: a plain `UNIQUE(vehicle_id, slot_start, slot_end)` would permanently block any future booking that reuses a slot after the original row is cancelled — directly violating AC5. `WHERE status = 'Booked'` was confirmed supported by pg-mem via a throwaway test and is standard Postgres syntax, so a cancelled/completed/no-show row no longer occupies its unique-index slot, letting a new booking reuse it immediately, with zero extra code.

**Honest limitation** (documented in the migration file's own comment too): the UNIQUE index only catches an **exact**-duplicate-slot race (both requests for literally the same `slotStart`/`slotEnd`). It does **not**, by itself, catch a DB-level race between two **partially**-overlapping-but-not-identical slots (e.g. 10:00–10:30 vs. 10:15–10:45) — that correctness rests on the app-level pre-check, which is not immune to a true multi-connection race on a real concurrent Postgres server (this sandbox's pg-mem harness cannot exercise or disprove that scenario either way, since it has no real concurrent connections). A real Postgres deployment **should** additionally add a `tsrange` + `EXCLUDE USING gist (vehicle_id WITH =, slot_range WITH &&) WHERE (status = 'Booked')` constraint for a true DB-enforced range-overlap guarantee, closing that residual gap — this was attempted here first and confirmed unusable in this sandbox (see point 3 above), so it could not be added or verified in this environment. This is the single most important honest caveat of this Story: **AC3/AC4's guarantee in a real multi-connection-concurrency Postgres deployment is stronger than what could be exercised or proven in this pg-mem sandbox**, but the exact-match backstop plus the general-overlap app-level check together are the strongest mechanism this sandbox could build and verify.

### AC5 status-exclusion proof required widening the CHECK constraint now (not deferred to #39)

`findBookedInRange`'s (and therefore the conflict-check's) `status = TEST_DRIVE_STATUS_BOOKED` filter is what makes "a cancelled booking frees its slot" true. Proving that with a real test requires actually writing a terminal status to a row — but `test_drives.status` had a DB CHECK constraint scoped to `IN ('Booked')` only (migration `1700000000013`), and a throwaway test confirmed **pg-mem does enforce CHECK constraints** (an `UPDATE ... SET status = 'Cancelled'` was rejected with `check constraint ... is violated`). Without widening it, the AC5 proof the orchestrator's brief explicitly asked for would be impossible to write. Migration `1700000000015` therefore widens the CHECK constraint to the full status vocabulary the entity already anticipated (`TEST_DRIVE_STATUS_COMPLETED`/`NO_SHOW`/`CANCELLED`, defined but unused since #34) — **one Story earlier than #34/#39's own comments anticipated**, but deliberately **only** the schema capability: no cancel/reschedule endpoint, DTO, or service method was built (that remains #39's job entirely). `test-drives.service.spec.ts`'s AC5 test proves the mechanism by writing the terminal status directly via `dataSource.query('UPDATE test_drives SET status = ...')`, bypassing the not-yet-built #39 endpoint exactly as instructed.

### Nearest-open-slot suggestion (AC2) — a small duplicated backend port of `schedulerGrid.ts`, not a shared package

`backend/src/test-drives/nearest-open-slots.ts#computeNearestOpenSlots` mirrors `frontend/src/utils/schedulerGrid.ts#computeDaySlots`'s fixed-grid + overlap-marking logic (same 09:00–19:00 UTC window, same 30-minute step) but is a separate, cross-referenced implementation — the backend (NestJS) and frontend (Vite/React) are separate packages with no existing shared-code mechanism, and standing one up for one ~15-line pure function was judged disproportionate for this fast-tracked Story. This mirrors #34/#35's own already-accepted precedent of duplicating the same operating-hours constants across both packages with a cross-reference comment. Scope limit: it only searches the **same calendar date** as the requested slot (may return fewer than the requested count, or none, near the end of the operating day) — acceptable per AC2's "suggests", not "guarantees", wording.

### Response shape — `{ errors, suggestedSlots }`, a deliberate departure from the plain `FieldError[]` array convention

Every other error in this codebase (`TestDriveEnquiryNotFoundError`, `OperatingHoursViolationError`, `LeadAlreadyConvertedError`, etc.) returns a bare `FieldError[]` JSON array. `TestDriveSlotConflictError` is the first error needing to carry structured data (the suggestion list) beyond field/message pairs, so its filter returns `{ errors: FieldError[], suggestedSlots: SuggestedSlot[] }` instead. `frontend/src/api/client.ts`'s `request()` was extended to parse both shapes (array-shaped bodies still populate `fieldErrors` exactly as before; the new object shape additionally populates `suggestedSlots`), so this is additive/backward-compatible, not a breaking change to any existing error path.

### Frontend conflict UX — in-form re-fill via `setValue`, not a query-param navigation

`NewTestDriveForm` shows the conflict as a form-level alert (not tied to one field — a vehicle/slot conflict is a cross-field business rule, not a single bad input) plus, when present, a row of clickable suggested-slot buttons. Clicking one calls `setValue('date', ...)`/`setValue('time', ...)` (mirroring the DSE's own visible fields) and clears the conflict message — the DSE reviews the refilled form and clicks "Book Test Drive" again, rather than the click silently auto-booking on their behalf. This is a simpler in-page re-fill, not #35's cross-page query-param pre-fill mechanism (`?vehicleId=&date=&time=`), since the DSE is already on this exact form when the conflict occurs — no navigation is needed.

## Known gaps / follow-ups for future Stories

- **No true DB-level range-overlap exclusion constraint** (GiST `EXCLUDE`) — confirmed unusable in this pg-mem sandbox (`btree_gist` extension unavailable). A real Postgres deployment should add `EXCLUDE USING gist (vehicle_id WITH =, slot_range WITH &&) WHERE (status = 'Booked')` as an additional defense-in-depth layer this Story's sandbox could not verify. This is the most important open item for a production hardening pass.
- **The partial UNIQUE index backstop only catches exact-duplicate-slot races**, not partial-overlap races, at the DB layer — see "Concurrency mechanism" above for the full honest accounting of what is and isn't guaranteed in this sandbox vs. a real concurrent Postgres deployment.
- **Nearest-open-slot suggestions are same-day only** (AC2) — a slot requested very late in the operating day may get zero or fewer than 3 suggestions; the 409 is still returned correctly, just with a shorter (or empty) suggestion list.
- **No location-matching requirement** between the selected vehicle and the actor — unchanged gap from #34, not this Story's concern.
- **No cancel/reschedule endpoint** — the CHECK constraint now legally allows `Completed`/`No-show`/`Cancelled`, but no controller/service method writes them; that remains #39's ("Log Test Drive Outcome") entire territory, this Story only proves the conflict-check correctly reacts to a terminal status once #39 (or a direct DB write, as in this Story's own test) sets one.
- No Playwright/E2E suite was written for this fast-tracked Story, mirroring every other fast-tracked Story in this codebase (#26, #29, #30, #31, #34, #35, etc.).
