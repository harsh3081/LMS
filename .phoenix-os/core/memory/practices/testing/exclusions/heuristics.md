# Heuristic Detection Rules

**Purpose**: Runtime detection rules for exclusion-worthy files that cannot be known statically from L1 defaults.
**Layer**: L2 (Heuristic Detection) in the four-layer exclusion resolution pipeline (#345).
**Consumed by**: `test-planner.md` Step 2.7 (Heuristic Exclusion Scan)
**Input**: `stackDescriptor.stackName` from `stack-detection.md` (schema 1.0)
**Output**: Array of glob patterns stored in `test-plan.json` under `heuristicExclusions` key (additive to L1 defaults)

**Relationship to other layers**:
- L1 (`defaults.md`): Static per-stack defaults â€” always applied; L2 results are additive, never duplicating L1
- L3 (`project-config-import.md`, #375): Project-level config imports â€” applied after L2
- L4 (`user-overrides.md`, #376): User overrides + merge algorithm â€” integrates all layers into final set

---

## Performance Budget

| Scan Type | Categories | Limit |
|-----------|-----------|-------|
| Content scanning | Cat 1 + Cat 2 | <=100 files total, first 10 lines per file |
| Glob matching | Cat 3 + Cat 4 | No file content reads required |

**Execution order**: Run Cat 3 + Cat 4 first (glob-only, instant), then Cat 1 + Cat 2 (content scan, shared 100-file budget).

**File selection for content scan**: Use targeted naming patterns first (e.g., `index.*` for barrels, `*.pb.go` for generated Go), then content-verify. Do NOT recursively scan vendor or dependency directories. Scope to project root and first-level `src/` (or equivalent) directories.

**Budget allocation between Cat 1 and Cat 2**: No hard split. Cat 2 naturally targets fewer files (named index files / `__init__.py` only). Cat 1 candidates are prioritized first; remaining budget is used for Cat 2.

## Category 1: Generated Code Markers

Detect files containing code generation markers in their first 10 lines. A file is exclusion-worthy if any of the first 10 lines matches a marker for the resolved stack or the `all` pseudo-key.

### Detection Rules

#### `all` (applies to every stack)

| Marker String | Line Match Type | Notes |
|---------------|-----------------|-------|
| `// Code generated` | starts-with or contains | Go convention, adopted broadly |
| `# This file is auto-generated` | starts-with or contains | Python/shell convention |
| `/* This file was generated` | starts-with or contains | Generic C-family convention |
| `// DO NOT EDIT` | starts-with or contains | Common generated-file warning |

#### `nodejs`

No additional markers beyond `all`.

#### `python`

No additional markers beyond `all`.

#### `go`

| Marker String | Line Match Type | Notes |
|---------------|-----------------|-------|
| `// Code generated by` | starts-with | Canonical Go tool convention |

> Note: `// Code generated by` is a specialization of the `all` rule `// Code generated`. Both are checked; a match on either is sufficient.

#### `java`

| Marker String | Line Match Type | Notes |
|---------------|-----------------|-------|
| `@Generated` | contains | javax.annotation.Generated (Java EE / Jakarta EE) |
| `@javax.annotation.Generated` | contains | Fully-qualified form |
| `@jakarta.annotation.Generated` | contains | Jakarta EE renamed form |

#### `dotnet`

| Marker String | Line Match Type | Notes |
|---------------|-----------------|-------|
| `[GeneratedCode` | contains | System.CodeDom.Compiler.GeneratedCodeAttribute |
| `// <auto-generated>` | contains | Standard .NET auto-generated XML comment |
| `// <auto-generated/>` | contains | Self-closing variant |

### Scan Strategy

1. Identify candidate files: source files matching stack extensions
   - `nodejs`: `*.ts`, `*.tsx`, `*.js`, `*.jsx`, `*.mjs`, `*.cjs`
   - `python`: `*.py`
   - `java`: `*.java`
   - `go`: `*.go`
   - `dotnet`: `*.cs`, `*.vb`, `*.fs`
2. Prioritize files in root `src/` or equivalent source directory; do not recurse into vendor directories
3. **File selection priority when >100 candidates exist**: (a) prefer files matching known generated-code naming patterns (`*generated*`, `*.pb.go`, `*.g.cs`), then (b) prefer files in `src/` root directory, then (c) alphabetical order. Truncate at 100 files; log: `"Heuristic scan limited to first 100 files out of {N} candidates"`
4. Cap candidate list at 100 files (shared with Cat 2; Cat 1 candidates are added first)
4. For each candidate: read first 10 lines only
5. If any line matches a marker string for the resolved `stackName` or `all`: emit the project-relative file path as a glob pattern
6. Before emitting: verify the pattern is not already present in L1 defaults for the resolved stack

### Output Format

Each matched file produces one glob entry: the project-relative file path.

Example: `src/api/generated-client.ts`

### Positive / Negative Scenarios

| Scenario | File | Stack | Expected |
|----------|------|-------|----------|
| Go generated file | `gen/model.pb.go` (line 1: `// Code generated by protoc`) | `go` | Match |
| Java generated class | `src/gen/UserMapper.java` (line 3: `@Generated`) | `java` | Match |
| .NET designer file | `Form1.Designer.cs` (line 1: `// <auto-generated>`) | `dotnet` | Match |
| Marker on line 11 | Any file with marker outside first 10 lines | any | No match (outside window) |
| Normal source file | `src/utils/format.ts` (no marker in first 10 lines) | `nodejs` | No match |

## Category 2: Barrel / Re-export Files

Detect index files whose content is predominantly export or re-export statements. These files add no testable logic and should be excluded from coverage requirements.

### Threshold Definition

A file is a barrel if: `(qualifying lines / total non-empty non-comment lines) > 0.90`

**Qualifying line** = a line that is an export or import/re-export statement per the stack rules below.
**Non-empty non-comment line** = a line that, after trimming whitespace, is non-empty and does not match the comment patterns for the stack.

The threshold of 90% is a fixed constant in this version. It is not parameterizable through config (future L4 override support may relax this).

### Detection Rules

#### `nodejs`

- **Candidate files**: `**/index.ts`, `**/index.tsx`, `**/index.js`, `**/index.jsx`
- **Threshold**: >90% of non-empty, non-comment lines are export statements
- **Export patterns** (qualifying lines): Lines starting with any of:
  - `export `
  - `export {`
  - `export default`
  - `export *`
  - `export type`
  - `module.exports`
- **Comment patterns** (excluded from ratio): Lines starting with `//`, `/*`, or `*`

#### `python`

- **Candidate files**: `**/__init__.py`
- **Threshold**: >90% of non-empty, non-comment lines are import/re-export statements
- **Import patterns** (qualifying lines): Lines starting with any of:
  - `from `
  - `import `
- **Comment patterns** (excluded from ratio): Lines starting with `#`

#### `java`

No barrel detection rules. Java does not use barrel/re-export patterns.

#### `go`

No barrel detection rules. Go does not use barrel/re-export patterns.

#### `dotnet`

No barrel detection rules. .NET does not use barrel/re-export patterns.

### Scan Strategy

1. Glob for candidate files per stack rules above
2. These candidates share the 100-file budget with Cat 1 (Cat 1 candidates are prioritized first)
3. Read first 10 lines of each candidate (sufficient for threshold computation in most barrel files)
4. Count qualifying lines and non-empty non-comment lines from the 10-line sample
5. If `qualifying / non_empty_non_comment > 0.90`, emit the file path as a glob pattern
6. Before emitting: verify the pattern is not already present in L1 defaults

> **Note on scan depth**: Reading only the first 10 lines may undercount exports in very long barrel files (e.g., a 500-line index.ts). This is intentional â€” the 10-line limit keeps performance within budget. Files that are true barrels almost always declare exports from the first line.

### Output Format

Each matched file produces one glob entry: the project-relative file path.

Example: `src/components/index.ts`

### Positive / Negative Scenarios

| Scenario | File | Stack | Expected |
|----------|------|-------|----------|
| Pure barrel (100% exports) | `src/index.ts` (10/10 qualifying lines) | `nodejs` | Match |
| Near-barrel (>90% exports) | `src/index.ts` (9/10 qualifying, 1 blank) | `nodejs` | Match |
| Below threshold (89%) | `src/index.ts` (8/9 qualifying out of 9 non-empty) | `nodejs` | No match |
| Python barrel | `pkg/__init__.py` (10/10 import lines) | `python` | Match |
| Mixed logic file | `src/index.ts` (5/10 qualifying) | `nodejs` | No match |

## Category 3: Auto-Generated Config / Cache

Static naming patterns matched by glob only. No file content reads are performed.

The agent matches these patterns against repository files and directories. If a match is found, the pattern is emitted. Deduplication against L1 defaults is applied before output.

### Detection Rules

#### `nodejs`

| Pattern | Description |
|---------|-------------|
| `.eslintcache` | ESLint cache file |
| `tsconfig.tsbuildinfo` | TypeScript incremental build info |
| `**/*.tsbuildinfo` | Nested TypeScript build info |
| `package-lock.json` | npm lock file |
| `yarn.lock` | Yarn lock file |
| `pnpm-lock.yaml` | pnpm lock file |
| `.npm/**` | npm cache directory |

#### `python`

| Pattern | Description |
|---------|-------------|
| `poetry.lock` | Poetry lock file |
| `Pipfile.lock` | Pipenv lock file |
| `**/*.pyc` | Compiled Python bytecode |
| `**/__pycache__/**` | Python bytecode cache directory |
| `.mypy_cache/**` | mypy type check cache |
| `.ruff_cache/**` | Ruff linter cache |

#### `java`

| Pattern | Description |
|---------|-------------|
| `**/*.class` | Compiled Java bytecode |
| `.gradle/**` | Gradle build cache |
| `**/target/**` | Maven build output |
| `**/.classpath` | Eclipse classpath file (auto-generated by IDE) |
| `**/.project` | Eclipse project file (auto-generated by IDE) |

#### `go`

| Pattern | Description |
|---------|-------------|
| `go.sum` | Go module checksum database |

#### `dotnet`

| Pattern | Description |
|---------|-------------|
| `**/obj/**` | .NET intermediate build output |
| `**/obj/**/*.json` | .NET build intermediate JSON files |
| `**/*.Designer.cs` | WinForms / WPF designer-generated C# (check L1 before emitting) |
| `**/*.g.cs` | Source generator output |
| `**/*.g.i.cs` | Source generator intellisense intermediates |

#### `all` (applies to every stack)

| Pattern | Description |
|---------|-------------|
| `.DS_Store` | macOS Finder metadata |
| `Thumbs.db` | Windows thumbnail cache |
| `**/*.bak` | Generic backup files |

### Output Format

Glob patterns as listed above. Agent emits only patterns NOT already present in L1 defaults for the resolved stack.

> **Glob-only constraint**: No file reads. The agent uses glob matching only to check if these patterns resolve to existing files or directories. Cat 3 shares no file scan budget.

## Category 4: Vendor / Dependency Directories

Static directory patterns matched by glob only. No file content reads are performed.

The agent verifies the directory exists in the repository before emitting the pattern. Deduplication against L1 defaults is applied before output.

### Detection Rules

#### `nodejs`

| Pattern | Description |
|---------|-------------|
| `node_modules/**` | npm / Yarn / pnpm dependencies |
| `bower_components/**` | Bower dependencies (legacy) |
| `packages/**/node_modules/**` | Monorepo workspaces â€” nested node_modules |

> **Monorepo note**: `packages/**/node_modules/**` should only be emitted if the repository contains a `packages/` directory (indicating a monorepo workspace layout). Check for directory existence before emitting.

#### `python`

| Pattern | Description |
|---------|-------------|
| `.venv/**` | Virtual environment (dotfile convention) |
| `venv/**` | Virtual environment (plain name convention) |
| `env/**` | Virtual environment (alternative naming) |
| `__pypackages__/**` | PEP 582 local packages directory |
| `.tox/**` | tox test environment directory |
| `site-packages/**` | Installed packages (when present in repo) |

#### `java`

| Pattern | Description |
|---------|-------------|
| `.m2/**` | Local Maven repository cache (if present in repo) |
| `.gradle/caches/**` | Gradle dependency cache (if present in repo) |

#### `go`

| Pattern | Description |
|---------|-------------|
| `vendor/**` | Go module vendor directory (`go mod vendor`) |

#### `dotnet`

| Pattern | Description |
|---------|-------------|
| `packages/**` | NuGet packages directory (legacy `packages/` restore) |
| `.nuget/**` | NuGet cache (if present in repo) |

### Output Format

Directory glob patterns as listed. Agent checks that the directory exists before emitting the pattern. An existing but empty directory still triggers the emit.

> **Glob-only constraint**: No file reads. Directory existence check is a filesystem probe only (equivalent to checking if the path resolves), not a content scan.

### Positive / Negative Scenarios

| Scenario | Stack | Pattern | Expected |
|----------|-------|---------|----------|
| `node_modules/` present in repo root | `nodejs` | `node_modules/**` | Match (emit) |
| `vendor/` present in Go project | `go` | `vendor/**` | Match (emit) |
| `.venv/` present in Python project | `python` | `.venv/**` | Match (emit) |
| `packages/` absent in dotnet project | `dotnet` | `packages/**` | No emit (dir not found) |
| `packages/` present in nodejs monorepo | `nodejs` | `packages/**/node_modules/**` | Match (emit if packages/ exists) |

## Agent Integration Instructions

### When to Run

After `test-planner.md` Step 2.6 (Emit Descriptor), before Step 3 (Pre-flight Validation). Runs unconditionally â€” no pre-condition check on project type or stack.

### Input

- `stackDescriptor.stackName` from `test-plan.json` (written by Step 2.6)
- Valid values: `nodejs`, `python`, `java`, `go`, `dotnet`
- Also apply the `all` pseudo-key rules for every stack

### Execution Order

1. Execute Cat 3 + Cat 4 (glob-only, no file reads) â€” apply patterns for `stackName` + `all`
2. Execute Cat 1 + Cat 2 (content scanning, shared 100-file budget) â€” read first 10 lines per candidate
3. Collect all matched patterns
4. Deduplicate against L1 defaults (`defaults.md` for resolved `stackName`)
5. Store results

### Output Contract

Store in `test-plan.json` under `heuristicExclusions` key:

```json
{
  "heuristicExclusions": {
    "generatedCode": ["src/api/generated-client.ts"],
    "barrels": ["src/components/index.ts"],
    "autoGenConfig": [".eslintcache", "tsconfig.tsbuildinfo"],
    "vendorDirs": ["node_modules/**"],
    "combined": ["src/api/generated-client.ts", "src/components/index.ts", ".eslintcache", "tsconfig.tsbuildinfo", "node_modules/**"]
  }
}
```

- `generatedCode`: Cat 1 results (file paths as globs)
- `barrels`: Cat 2 results (file paths as globs)
- `autoGenConfig`: Cat 3 results (named patterns)
- `vendorDirs`: Cat 4 results (directory patterns)
- `combined`: Flat union of all four categories â€” this is the L2 input consumed by #376 merge algorithm
- All values are glob pattern strings, deduplicated against L1 defaults
- **No-op behavior**: When nothing matches, all arrays are `[]` â€” downstream behavior is identical to pre-L2

### Logging

- On success: `"Heuristic scan complete: {N} additional exclusion patterns discovered"`
- On empty result: `"Heuristic scan: no additional patterns found"`
- On file-not-found: `"heuristics.md not found, skipping heuristic scan"` (set all arrays to `[]`, continue)

### Additive-Only Constraint

Heuristic results are strictly additive. They supplement L1 defaults but never remove or override them. The `combined` array is a union of new patterns only (L1 defaults are not re-emitted here; they are added by the #376 merge algorithm).

### Performance Budget Enforcement

| Check | Limit | Action on Violation |
|-------|-------|---------------------|
| Content scan file count (Cat 1 + Cat 2) | <=100 files | Truncate candidate list at 100; log warning |
| Lines read per file | First 10 lines only | Read exactly 10 lines (or EOF if file is shorter) |
| Glob matching (Cat 3 + Cat 4) | No file reads | Use filesystem glob/directory existence checks only |

## STM Storage Format

The `heuristicExclusions` key is additive in `test-plan.json`. Existing consumers that do not recognize this key will ignore it without error, preserving backward compatibility.

### Schema

```json
{
  "stackDescriptor": { "...": "..." },
  "heuristicExclusions": {
    "generatedCode": ["path/to/generated-file.go", "path/to/Generated.cs"],
    "barrels": ["src/components/index.ts"],
    "autoGenConfig": [".eslintcache", "tsconfig.tsbuildinfo"],
    "vendorDirs": ["node_modules/**"],
    "combined": [
      "path/to/generated-file.go",
      "path/to/Generated.cs",
      "src/components/index.ts",
      ".eslintcache",
      "tsconfig.tsbuildinfo",
      "node_modules/**"
    ]
  }
}
```

### Empty-Result Schema (no-op case)

```json
{
  "heuristicExclusions": {
    "generatedCode": [],
    "barrels": [],
    "autoGenConfig": [],
    "vendorDirs": [],
    "combined": []
  }
}
```

### Downstream Consumer

`heuristicExclusions.combined` is the L2 input consumed by the #376 merge algorithm. The per-category arrays (`generatedCode`, `barrels`, `autoGenConfig`, `vendorDirs`) are retained for traceability and debugging.
