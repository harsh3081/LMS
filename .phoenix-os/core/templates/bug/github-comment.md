# GitHub Comment Template for Bug Analysis

This template defines the structured format for posting Bug Analysis results as GitHub issue comments.

## Template Variables

- `{issue-number}`: Bug issue number
- `{bug-title}`: Bug issue title
- `{confidence-level}`: Confidence level (High/Medium/Low)
- `{severity}`: Bug severity (Critical/High/Medium/Low)
- `{root-cause-summary}`: One-line summary of root cause
- `{root-cause-details}`: Detailed root cause explanation
- `{affected-files}`: List of affected files
- `{fix-recommendations}`: Summary of required changes
- `{test-requirements}`: Summary of required tests

## Output Format

```markdown
## Bug Analysis Complete

**Issue**: #{issue-number} - {bug-title}
**Confidence**: {confidence-level}
**Severity**: {severity}

---

### Root Cause

{root-cause-summary}

<details>
<summary>Detailed Analysis</summary>

#### Why it broke
{root-cause-details}

#### Where it broke
{affected-files}

</details>

---

### Recommended Fix

{fix-recommendations}

### Required Tests

{test-requirements}

---

### Apply Fix

Run the following command to implement the fix:

```bash
/phoenix:build:fix-bug {issue-number}
```

---

Generated by Phoenix OS Bug Analysis Workflow
```

## Formatting Rules

### Root Cause Summary
- Keep to one line (max 100 characters)
- Focus on the "why" not the "what"
- Use clear, non-technical language where possible

### Detailed Analysis
- Wrap in `<details>` tag to keep comment concise
- Include technical details for developers
- Reference specific code locations

### Fix Recommendations
- Keep actionable and specific
- Number recommendations if multiple
- Reference specific files and changes

### Test Requirements
- List as bullet points
- Specify test type (unit, integration, e2e)
- Keep descriptions concise

## Implementation Notes

1. **Collapsible Sections**: Use `<details>` for lengthy content to keep comments scannable
2. **Command Reference**: Always include the apply-fix command
3. **Attribution**: Include Phoenix OS attribution footer
4. **Markdown Rendering**: Ensure proper rendering in GitHub UI
5. **Mobile Friendly**: Keep visible content concise for mobile viewing
