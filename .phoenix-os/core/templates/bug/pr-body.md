# Pull Request Body Template for Bug Fixes

This template defines the structured format for bug fix pull request descriptions.

## Template Variables

- `{issue-number}`: Bug issue number
- `{bug-title}`: Bug issue title
- `{root-cause-summary}`: One-line summary of root cause
- `{files-changed}`: List of files modified
- `{changes-summary}`: Summary of changes made
- `{test-coverage}`: Tests added or modified
- `{quality-gates}`: Quality gate results
- `{risk-level}`: Risk level (Low/Medium/High)
- `{implementation-notes}`: Additional implementation notes

## Output Format

```markdown
## Bug Fix for Issue #{issue-number}

Fixes #{issue-number}

### Root Cause Analysis

**Why it broke:**
{root-cause-summary}

**Where it broke:**
{files-changed}

**What was fixed:**
{changes-summary}

### Changes Made

{changes-summary}

### Test Coverage

{test-coverage}

**Test Summary:**
- New tests added: {count}
- Tests modified: {count}
- All tests passing: Yes/No

### Quality Gates

{quality-gates}

- [ ] All tests passing
- [ ] Linting passed
- [ ] Build successful
- [ ] No new security vulnerabilities

### Impact

**Risk Level:** {risk-level}

{implementation-notes}

---
Generated by Phoenix OS Bug Fix Workflow

Co-Authored-By: Claude <noreply@anthropic.com>
```

## Formatting Rules

### Issue Reference
- Use `Fixes #` syntax for automatic issue linking
- GitHub will close the issue when PR is merged

### Root Cause Summary
- Keep concise but complete
- Reference the analysis from RCA
- Focus on developer understanding

### Changes Made
- List files changed with brief descriptions
- Group related changes together
- Highlight critical changes

### Test Coverage
- List new tests added
- Note any modified tests
- Include test counts

### Quality Gates
- Use checkboxes for visual status
- Include all gates that were run
- Note any warnings or issues

### Risk Level
- **Low**: Simple, isolated change
- **Medium**: Moderate scope, well-tested
- **High**: Complex change, requires careful review

## Implementation Notes

1. **Auto-linking**: Use `Fixes #` for automatic issue closure
2. **Co-authorship**: Include Claude co-author credit
3. **Attribution**: Include Phoenix OS footer
4. **Completeness**: All sections should have content
5. **Reviewability**: Provide enough context for code review
