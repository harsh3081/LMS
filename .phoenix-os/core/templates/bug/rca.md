# Root Cause Analysis Template

This template defines the structured format for Bug Root Cause Analysis (RCA) output.

## Template Variables

- `{issue-number}`: Bug issue number
- `{bug-title}`: Bug issue title
- `{analysis-date}`: Analysis timestamp
- `{root-cause-explanation}`: Detailed explanation of why the bug occurred
- `{affected-files}`: List of affected files with line numbers
- `{required-changes}`: Specific changes needed to fix the bug
- `{test-coverage-required}`: Required test cases
- `{confidence-level}`: Confidence level of analysis (High/Medium/Low)

## Output Format

```markdown
## 🔍 Bug Analysis - Root Cause Analysis

**Issue**: #{issue-number} - {bug-title}
**Analysis Date**: {analysis-date}
**Confidence Level**: {confidence-level}

### Root Cause Analysis

#### Why it broke
{root-cause-explanation}

Key factors:
- Factor 1: {details}
- Factor 2: {details}
- Factor 3: {details}

#### Where it broke
{affected-files}

File-by-file breakdown:
- **File**: `{file-path}:{line-number}`
  - **Function**: `{function-name}`
  - **Issue**: {specific-issue}
  - **Current behavior**: {current-behavior}
  - **Expected behavior**: {expected-behavior}

#### What needs to be fixed
{required-changes}

Specific changes required:
1. **Change 1**: {description}
   - **File**: `{file-path}`
   - **Action**: {add/modify/remove}
   - **Details**: {implementation-details}

2. **Change 2**: {description}
   - **File**: `{file-path}`
   - **Action**: {add/modify/remove}
   - **Details**: {implementation-details}

### Test Coverage Required
{test-coverage-required}

Required test cases:
- **Test 1**: {scenario-description}
  - **Type**: {unit/integration/e2e}
  - **Assertion**: {what-to-verify}

- **Test 2**: {edge-case-description}
  - **Type**: {unit/integration/e2e}
  - **Assertion**: {what-to-verify}

- **Test 3**: {regression-check-description}
  - **Type**: {unit/integration/e2e}
  - **Assertion**: {what-to-verify}

### Recommended Fix Approach

{fix-approach-description}

### How to Apply Fix

Run the following command to automatically implement and test the fix:

```bash
/phoenix:build:fix-bug {issue-number}
```

---
🤖 Generated by Phoenix OS Bug Analysis
```

## Formatting Rules

### Root Cause Explanation
- Start with high-level explanation
- Provide specific technical details
- Reference code snippets if helpful
- Explain assumptions that were violated

### Affected Files
- Use absolute or repository-relative paths
- Include line numbers for precise location
- Specify function/method names
- Describe the specific issue in each location

### Required Changes
- Number changes sequentially
- Specify exact file paths
- Use clear action verbs (add, modify, remove, refactor)
- Provide enough detail for implementation

### Test Coverage
- List all scenarios that need testing
- Include happy path, edge cases, and regressions
- Specify test type (unit, integration, e2e)
- Describe what each test should verify

### Confidence Level
- **High**: Root cause clearly identified with strong evidence
- **Medium**: Root cause identified but requires validation
- **Low**: Multiple possible root causes, needs more investigation

## Implementation Notes

1. **Variable Substitution**: Agent should replace all `{variable}` placeholders with actual values
2. **Code References**: Use backticks for code elements (files, functions, variables)
3. **Markdown Formatting**: Preserve markdown syntax for proper rendering
4. **Line Breaks**: Use appropriate spacing for readability
5. **Specificity**: Be as specific as possible with file paths and line numbers
